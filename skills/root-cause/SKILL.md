# Root Cause Analysis

ユーザの問題・要望に対して、表面的な解決ではなく根本原因を特定し、複数の解決策とトレードオフを提示する。

## 手順

1. 問題の特定: 会話コンテキストから問題・要望を把握する。スコープ（直近のやり取りか会話全体か）は問題の性質に応じて判断。
2. 根本原因の分析: なぜその問題が起きているのか深掘りする。必要に応じてコードベースを調査（Grep/Read等）して裏付けを取る。
   - 深掘りの停止条件: 「次の why に対する打ち手が、ユーザの統制範囲・工数で実行可能」な最深層で止める。具体的には 2〜3 段が目安。
   - 浅すぎる例: 「ORM の lazy load を呼んでいる」（個別バグ修正しか出てこない）
   - 適切な例: 「CI に N+1 検知機構がない」（チームで導入可能、施策が複数浮かぶ）
   - 深すぎる例: 「組織がパフォーマンスを後回しにする文化」「評価制度が悪い」（経営判断・人事制度変更が必要で、ユーザの権限で実行不能）
   - 判定基準: 一段下の why に対し具体的施策が 2 個以上思い浮かぶか。思い浮かばなければ深すぎ、1 個しか出ないなら表面的。
3. 解決策の列挙: 表面解と根本解をそれぞれ独立に列挙する。
4. トレードオフ比較: 問題の性質に応じた比較軸を選び、表形式で比較する。
5. 推奨の提示: 最も推奨する案に ★ マークを付ける。

## 出力フォーマット

以下のセクション構成で出力する。

```
## 根本原因
{なぜこの問題が起きているのか}

## 表面解
- 方法: ...
- メリット: ...
- デメリット: ...

## 根本解A
- 方法: ...
- メリット: ...
- デメリット: ...

## 根本解B（ある場合）
- 方法: ...
- メリット: ...
- デメリット: ...

## トレードオフ比較
| | 表面解 | 根本解A | 根本解B |
|---|---|---|---|
| {軸1} | ... | ... | ... |
| {軸2} | ... | ... | ... |

組み合わせ: {複数案が組み合わせ可能なら「A+Bで〇〇も可能」等を明記。なければこの行ごと省略}

★ 推奨: {推奨案 (単一案 or 組み合わせ案 どちらも可) とその理由。他の根本解を推奨しなかった理由も明示する}
```

## ルール

- 分析のみ: 実装は行わない。ユーザが案を選んだ後に別途実装する。
- 案の独立性: A案とB案を混ぜてC案としない。各案は独立した解決策として提示する。
- 組み合わせ可能性: 複数の案が組み合わせ可能な場合、トレードオフ比較の直後（★推奨より前）に「組み合わせ: A+Bで〇〇も可能」と明記する。
- 案の数に制限なし: 有効な根本解が複数あればすべて列挙する。
- 比較軸はAI判断: 問題の性質に応じて最も重要な比較軸を選ぶ（例: 工数、再発リスク、影響範囲、副作用、保守性、パフォーマンス等）。
- 同調より正確さ: ユーザーの前提・仮説・希望する方向性と矛盾しても、根拠に基づく結論を曲げない。「ユーザーが望む答え」ではなく「正確な分析」を出す。ただし異論には必ず根拠を添える。
- コード調査: 以下の判断基準で要否を決め、スキップする場合はその旨を「根本原因」セクション冒頭に 1 行で明記する。
  - 必要: 問題が特定コードベースの動作・実装に紐づく場合（バグ・パフォーマンス・設計・依存関係）
  - 不要: 組織/プロセス問題、一般論レベルの設計相談、コードベースが提示されていない場合
  - 必要と判断した場合は Grep/Read で裏付けを取ってから分析する
