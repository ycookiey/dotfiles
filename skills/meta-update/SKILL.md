# meta-update

セッション中の学びを永続化するskill。CLAUDE.mdの「自己改善」シグナル検知時に起動される、または手動起動。引き金の判定はCLAUDE.md側、抽出・ルーティング・承認・書込はこのskillの責務。

## 原則

- **提案のみ**: 勝手に書かない。記録先と内容を提示し承認を得てから反映
- **コード・命名は変更しない**: 誤誘導の原因がコード構造/関数名/コメントの場合、改善は提案テキストに留める(プロダクト資産のため)
- **遅延コミット**: 投機的に記録しない。実需(再発・明示要求)が確認できた学びのみ

## Phase 1: 抽出と除外

会話を走査し、永続化すべき学びを1行知識(暗黙知を明文化)に圧縮する。各学びにシグナル(再作業/誤誘導/行動指針/陳腐化)を対応づける。陳腐化は新規追記でなく**既存記述の修正・置換**が成果物(矛盾する注記の追記でなく古い記述自体を直す)。

次を**順に**判定し、1つでも該当すれば除外する:

1. 一回性の事実・特定対象固有の解釈・その場限りの訂正 → 除外(一般則化できない)
2. 別agentテスト: 「同じ情報で別agentも同じ判断/誤解をするか」が NO → 自分固有のミスなので除外
3. 既知の自明則(「READMEを読む」等) → 新規の学びを生まないので除外

残った学びの記録対象範囲を判定する: 当該skillの内部手順としてのみ必要な知識はそのskill内に留める。一方、**meta-update を起動しない通常作業でも遭遇する事象**は「全作業横断知識」として永続化対象とする。docs化の前に既存記載と重複しないか必ず確認する(skill内蔵知識との二重化を避ける)。

## Phase 2: ルーティング判定

### Step 1: スコープ判定(置き場所を決める)

「他プロジェクトでも成り立つ知識か」で判定:

- YES(環境横断) → `~/.claude` 配下(下表へ)
- NO(特定プロジェクト固有) → 対象プロジェクトの root / そのプロジェクトの `CLAUDE.md`。**先に対象パスを特定**(cwd と対象プロジェクトが異なる場合に注意)

### Step 2: 性質判定(環境横断知識の記録先)

| 学びの性質 | 記録先 |
|---|---|
| 手順・ハマりどころ・ツール仕様・調査結果 | docs |
| 繰り返す定型作業フロー | skill(新規作成 or 既存skill更新) |
| 全体方針・判断基準・参照先の追加 | CLAUDE.md |
| コード構造・命名・コメントの誤誘導 | **提案のみ**(書込まず改善案を提示) |

### Step 3: ファイル特定(docs/skill 行きの場合)

カテゴリ決定後、ファイル粒度まで降ろす: 既存 docs/skill を Glob → トピック一致を Grep → 一致あれば**追記**、なければ新規作成。重複記載を避ける。

判断に迷う場合は AskUserQuestion で記録先を確認。

## Phase 3: 承認

記録先・追記内容(diff相当)を提示し、AskUserQuestion で承認を取る。複数の学びがあれば個別に可否を選べる形にする。

## Phase 4: 書込

承認された学びのみ反映:

- docs: 高密度日本語フォーマット(skill `doc-optimizer` 準拠 — カタカナ→英語キーワード、述語省略、箇条書き)
- CLAUDE.mdへの追記は編集ルール遵守(簡潔・箇条書き・長文禁止)
- 既存docsに該当があれば新規作成せず追記

## 書込先のsymlink注意(重要)

`~/.claude` 配下はsymlinkでWrite拒否される。書込前に `readlink -f <対象>` で実体パス(dotfiles配下)を解決してからEdit/Writeする。実体マッピングは `~/.claude/docs/symlinks.md` 参照。
